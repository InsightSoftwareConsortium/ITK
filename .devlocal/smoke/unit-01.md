# Smoke Unit 01 — ImageIterator / ImageConstIterator

## Summary

Refactored the `ImageConstIterator` class hierarchy into a single
`ImageIteratorBase<TImage, bool VIsConst>` class template living in
`itkImageConstIterator.h`. Pointer fields (`m_Buffer`, `m_Image`) use
`std::conditional_t<VIsConst, const X*, X*>` so the non-const flavor no
longer needs `const_cast` to read through them.

Legacy names are preserved as alias templates:

```cpp
template <typename TImage>
using ImageConstIterator = ImageIteratorBase<TImage, true>;

template <typename TImage>
using ImageIterator = ImageIteratorBase<TImage, false>;
```

Files changed:

- `Modules/Core/Common/include/itkImageConstIterator.h` — replaced
  `class ImageConstIterator` with `class ImageIteratorBase<TImage,
  VIsConst>` and appended the `ImageConstIterator` alias.
- `Modules/Core/Common/include/itkImageIterator.h` — reduced to a thin
  header that `#include`s the base and defines the `ImageIterator`
  alias.
- `Set()` migrated from `ImageIterator` into the base, SFINAE-gated on
  `!VIsConst` via `template <bool VCopy = VIsConst,
  std::enable_if_t<!VCopy, int> = 0>`.
- `Value()` return type is now
  `std::conditional_t<VIsConst, const PixelType&, PixelType&>`.
- `GetImage()` return type is `ImagePointer =
  std::conditional_t<VIsConst, const ImageType*, ImageType*>`.
- Converting constructor (non-const → const) added on the base, gated by
  `VIsConst && !VOtherIsConst`, to preserve the previous
  `ImageIterator(const ImageConstIterator&)` implicit conversion path in
  the opposite direction (const-widening).

Public API preserved: `Set`, `Get`, `Value`, `GetImage`, `ComputeIndex`,
`SetIndex`, `SetRegion`, all comparisons, and all typedefs.

## Pitfalls

1. **`itkImageIterator.hxx` is orphaned.** The legacy `.hxx` defines
   out-of-class constructors and `operator=` for
   `ImageIterator<TImage>`. Alias templates cannot have out-of-class
   member definitions — those definitions would reference a class
   template that no longer exists under that name. The new
   `itkImageIterator.h` no longer `#include`s the `.hxx`. The `.hxx`
   file itself was left on disk (task constraints limited edits to the
   two listed headers). In the real refactor the `.hxx` must be
   deleted.

2. **Downstream iterator subclasses** (`ImageRegionConstIterator`,
   `ImageRegionIterator`, `ImageScanlineConstIterator`,
   `ImageScanlineIterator`, `ImageLinear*IteratorWithIndex`,
   `ImageRandom*`, etc.) inherit via
   `public ImageConstIterator<TImage>` / `public ImageIterator<TImage>`
   and use `using Superclass = ImageConstIterator<TImage>;`. Alias
   templates are transparent in that position (C++17), so these
   continue to work — the `Superclass::IndexType` etc. using-decls
   resolve to the aliased `ImageIteratorBase<TImage, true>`.

3. **`ImageIterator(const ImageConstIterator&)` constructor** (used in
   some convenience paths) does not survive as-is. The smoke refactor
   adds a converting ctor on the base that goes *const-widening*
   (non-const → const), which is the direction actually needed by
   downstream code (e.g. passing a mutable iterator where a const one
   is expected). The legacy *narrowing* direction (const → mutable) was
   an accident of the pre-refactor inheritance layout and was only
   usable because of the `const_cast` that this refactor removes. A
   real refactor must audit callers for that direction and either
   remove them or replace them with explicit `const_cast`s at the call
   site (after justifying each one).

4. **ITK_TEMPLATE_EXPORT on alias templates.** The alias template does
   not carry the storage-class attribute; the underlying
   `ImageIteratorBase` class template does. Explicit instantiations
   elsewhere in the tree (search for
   `template class ...ImageConstIterator<`) would need to target the
   base-with-bool form. A quick grep finds none in ITK proper.

5. **Python wrapping.** All `wrapping/` entries are `.notwrapped` —
   `itkImageConstIterator.notwrapped`, `itkImageIterator.notwrapped`,
   etc. No SWIG/castxml update required.

6. **CTAD.** The original deduction guide was
   `ImageConstIterator(SmartPointer<TImage>, const RegionType &) ->
   ImageConstIterator<std::remove_const_t<TImage>>;`. Alias templates
   do get deduction guides in C++20, but ITK currently targets C++17.
   To keep CTAD working, the guide now lives on
   `ImageIteratorBase` directly, and two guides are provided — one
   with `SmartPointer` → `VIsConst=true`, one with raw `TImage*` →
   `VIsConst=false`. Callers using `ImageIterator it{img, region};`
   CTAD will still need a C++20 mode or explicit template args — this
   is a potential regression that a real refactor must verify against
   the test suite.

## Recommendation for the real refactor

1. **Include the `.hxx` deletion.** Remove
   `Modules/Core/Common/include/itkImageIterator.hxx` and audit for
   stray `#include "itkImageIterator.hxx"` hits.
2. **Audit `ImageIterator(const ImageConstIterator&)` callers.** Use
   `git grep -n 'ImageIterator[^a-zA-Z_].*ImageConstIterator'` across
   the ITK tree plus the major consumers (BRAINSTools, Slicer,
   ANTs) — this conversion direction was always suspect.
3. **Validate CTAD fallout.** Run the full ITK test build (at minimum
   Modules/Core/Common tests, and the ImageIterator/ImageRegion
   iterator tests); CTAD regressions show up as unhelpful "cannot
   deduce" errors at existing call sites.
4. **Consider doing the same refactor on
   `ImageConstIteratorWithIndex` / `ImageIteratorWithIndex` and the
   ImageRegion-/Scanline-/Linear- flavors in the same patch series.**
   They all follow the identical `const_cast`-in-the-derived pattern
   and would benefit from the same VIsConst consolidation.
5. **Keep `Self = ImageIteratorBase` internal.** Subclasses pin
   `Superclass = ImageConstIterator<TImage>` (the alias) — that's
   fine. But any reflection-style code or serialization that uses
   `GetNameOfClass()` on the base will now get
   `"ImageIteratorBase"`. Verify no tests lock to the old name.
6. **Before pushing for real review**, open a Discourse thread
   summarizing the VIsConst pattern (already in use for
   `NeighborhoodIteratorBase`, `ShapedNeighborhoodIteratorBase`, and
   the new `ShapedFloodFilledFunctionConditionalConstIterator`) to get
   community buy-in on extending it systematically across the iterator
   hierarchy.
